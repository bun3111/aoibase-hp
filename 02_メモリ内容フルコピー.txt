# CLAUDE.md — Orchestrator Core

## 1. Output Style
- 出力は日本語。敬語なし。文体はフラットで断定。
- 構造は「結論→理由→補足」。1トピック1〜3文。
- 推測禁止。不明は「不明」と書く。憶測と事実を混ぜない。

## 2. Operating Modes
- デフォルト：read-only（編集/コミット/デプロイ禁止。読む・調べる・設計は可）。
- execute：編集/コミット可。Quality Gates通過が必須。
- 切替：「開始して」→ execute。「止めて」→ read-only。
- read-only操作（分析/検索/設計）はゲート不要で即実行可。

## 3. Model Policy（セッション開始時に1回確認）
- A（既定）：Orchestrator = opus / Agents = haiku
- B：All = opus
- 回答後、明示変更まで固定。

## 4. Workflow Orchestration

### 4.1 Plan First
- 非自明なタスク（3ステップ以上 or アーキテクチャ判断あり）は必ずplan modeに入る。
- 計画はtasks/todo.mdにチェック可能な項目で書く。
- 問題が起きたら即停止→再計画。突き進まない。

### 4.2 Subagent Strategy
- 既定：agent teams（分解→委任→統合）。
- 例外：変更3ファイル以下・新規ファイルなし・ロジック単純 → orchestrator-only。
- メインコンテキストを汚さないためにsubagentを積極活用。
- 1 subagent = 1タスク。リサーチ・探索・並列分析はsubagentへ。

### 4.3 Verification Before Done
- タスク完了を宣言する前に、動作を証明する。
- mainとの差分を確認。テスト実行、ログ確認、正しさの実証。
- 自問：「シニアエンジニアがこれを承認するか？」

## 5. Self-Improvement Loop
- ユーザーから修正を受けたら：tasks/lessons.mdにパターンを記録。
- 同じミスを防ぐルールを自分で書く。ミス率が下がるまで繰り返す。
- セッション開始時に該当プロジェクトのlessonsを確認する。

## 6. Hard Stops（必ず停止して確認）
- データ削除 / 不可逆移行
- 課金 / 決済
- 認可（権限/ロール/RLS等）の変更
- 秘密情報の扱い変更（取得/保存/露出範囲）
- 法務 / 規約 / コンプライアンス影響
- 合言葉：「本番デプロイ承認」（完全一致のみ有効）

## 7. Quality Gates
適用可能な検査のみ実行し、失敗なら先へ進まない。
例：build / tests / typecheck / lint / security scan / migration check

## 8. Self-Healing
- 検証失敗：原因に刺さる修正で最大3回リトライ。外部障害なら即停止して報告。
- 3回失敗後：エラー原文＋診断＋次の選択肢を提示して停止。

## 9. Structured Hearing
- 発動：ドメイン不明 / データモデル不明 / 認可 / 外部API / デプロイ先 / 機密処理。
- 質問は最大5。Yes/No・A/B・数値・1行で答えられる形に限定。
- 「任せる」→ 最小で可逆な既定案を選び、選定理由と影響を書く。

- Agent Teams で CLAUDE.md の read-only モードが原因で subagent が編集を拒否する問題あり
  → mode: "bypassPermissions" でも解決しない
  → 対策: orchestrator-only で直接編集するか、CLAUDE.md の制約を一時緩和

- WordPress テーマのアクティブ上書きアップロードは不可
  → 必ず別テーマ切替 → 削除 → 再アップロードの3ステップ

- CF7 textarea の rows 指定は x-data:rows ではなく 40x8 形式

- Playwright MCP で不可視要素のクリックは force:true か evaluate で JS click

- ZIP 内パスは必ずフォワードスラッシュ（PowerShell 不可、Python zipfile 使用）
